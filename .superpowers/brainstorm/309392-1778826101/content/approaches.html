Two approaches to the debug page

Both give you per-monitor detail + global feed + log capture. The difference is where logs live in the DB.

A

Log blob on the runs row — Recommended

Add a log TEXT column to the existing runs table. Runner captures ctx.logger output into a string buffer and writes it alongside the run result. Simple schema change, no joins needed.

Pros

  • One table, one query
  • Trivial to add
  • Fits current design

Cons

  • No per-line timestamps
  • Harder to filter by log level
B

Separate run_logs table

New table: run_logs(id, run_id, level, message, logged_at). Each log line is a row, linked to the run by run_id. More structured but requires a join on every detail page load.

Pros

  • Per-line timestamps
  • Filterable by level

Cons

  • Extra table + join
  • More schema complexity
  • Overkill for this use case
